# PicMind Phase 3 设计规格

## 目标

PicMind Phase 3 的目标是在 Phase 2 图片识别闭环之上，提升图库检索能力，并为后续真实视觉模型 Provider 接入预留配置骨架。

本阶段聚焦「智能图库检索与 Provider 配置骨架」。用户应该能通过关键词、标签、格式和排序快速定位图片；同时，设置页应明确展示当前识别 Provider 为 `mock`，并说明未来可替换为 Ollama 或云端视觉模型。

### 已完成

- 本地图片扫描、SQLite 索引和图片元数据持久化。
- 图库、图片详情、看板和设置页面。
- Mock 图片识别 Provider。
- 单张图片重新识别。
- 按 `image_ids` 创建批量识别任务。
- 批量识别状态查询。
- 识别结果持久化到 `annotations` 表。

### 验证状态

- 后端测试：`cd backend && uv run pytest -q`，44 passed。
- 前端构建：`npm --prefix frontend run build`，构建通过。

### 已知短板

- 识别结果仍来自 Mock Provider。
- 图库缺少关键词搜索、格式筛选和排序控制。
- 批量识别 API 当前创建后同步执行完成，进度查询更接近结果查询，不是真正的后台队列。

## 范围

### 包含

- 图库页增加搜索框，支持按 `caption`、`tags` 和 `file_path` 做关键词搜索。
- 图库页增加标签筛选、格式筛选和排序控件。
- 后端扩展图片列表 API，让搜索、筛选和排序在服务端完成。
- 设置页增加「识别 Provider」展示卡片。
- Provider 配置本阶段只读展示，默认值为 `mock`。
- 保持 Phase 2 的单张识别和批量识别功能可用。

### 不包含

- 不接入真实 Ollama 或云端视觉模型。
- 不做 embedding、向量搜索或自然语言问答。
- 不做批量任务持久化、取消或重试。
- 不做多用户权限或远程同步。
- 不新增标签聚合 API。

## 后端 API

### 图片列表 / List Images

- **请求方式 (Method):** GET
- **请求路径 (Path):** `/api/images`

#### 新增查询参数

| 参数名 (Field) | 类型 (Type) | 必填 (Required) | 说明 (Description) |
| --- | --- | --- | --- |
| q | string | 否 | 关键词搜索，匹配 `caption`、`tags` 和 `file_path`。 |
| tag | string | 否 | 按单个标签筛选，保留现有语义。 |
| format | string | 否 | 按图片格式筛选，例如 `JPEG`、`PNG`。 |
| sort | string | 否 | 排序字段，支持 `indexed_at`、`modified_at`、`file_size`、`width` 和 `height`。 |
| order | string | 否 | 排序方向，支持 `asc` 和 `desc`。 |

分页参数 `page` 和 `size` 保持不变。

#### 响应

响应结构保持现有 `ImageList` 不变：

```json
{
  "items": [],
  "total": 0,
  "page": 1,
  "size": 50
}
```

### 设置 / Get Settings

- **请求方式 (Method):** GET
- **请求路径 (Path):** `/api/settings`

响应继续包含：

- `watch_folders`
- `db_path`
- `provider`

本阶段将 `provider` 明确解释为当前识别 Provider 名称，默认值为 `mock`。设置页只展示状态和说明，不提供切换操作。

## 数据流

1. 用户在图库页输入关键词，或选择标签、格式和排序条件。
2. 前端更新筛选状态，并调用 `GET /api/images`。
3. 前端将筛选条件序列化为查询参数。
4. 后端根据参数组合数据库查询。
5. 后端返回分页后的图片列表和总数。
6. 前端展示结果数量、当前筛选条件和清空筛选入口。

## 前端设计

### 图库页

图库页顶部新增「检索工具栏」。

第一行包含：

- 搜索框。
- 搜索按钮或按 Enter 搜索。
- 清空筛选按钮。

第二行包含：

- 标签筛选。
- 格式筛选。
- 排序字段选择。
- 升序 / 降序选择。

右侧保留 Phase 2 的「已选择数量」和「批量识别」操作。

#### 交互要求

- 搜索框按 Enter 触发请求，避免每输入一个字就请求后端。
- 标签筛选先使用当前加载结果中出现的标签作为下拉选项。
- 格式筛选先使用当前加载结果中出现的格式作为下拉选项。
- 默认排序为 `indexed_at desc`。
- 搜索或筛选条件变化后刷新图库列表。
- 筛选条件变化后清空已选择图片，避免批量识别隐藏图片。
- 筛选无结果时显示「没有匹配的图片」，并提供「清空筛选」。
- 进入图片详情再返回时，保留当前图库筛选状态。

### 设置页

设置页新增「识别 Provider」卡片。

卡片内容包括：

- 当前 Provider：`mock`。
- 当前阶段说明：本地 Mock 识别用于验证流程。
- 后续扩展说明：未来可替换为 Ollama 或云端视觉模型。
- 明确提示：Phase 3 不支持在页面切换 Provider。

## 后端实现边界

- 图片查询逻辑继续放在现有 images API 或查询层，不新建独立搜索服务。
- `q` 搜索匹配：
  - `images.file_path`
  - `annotations.caption`
  - `annotations.tags`
- `format` 直接匹配 `images.format`。
- `tag` 继续匹配 `annotations.tags` 中的标签 JSON 字符串。
- `sort` 和 `order` 必须使用白名单校验。
- 不能把用户输入直接拼接进 SQL。
- 查询继续保留分页，避免大图库一次性返回全部图片。

## 错误处理

- 未知 `sort` 字段返回 400。
- 未知 `order` 值返回 400。
- 空 `q` 等同于不搜索。
- 搜索或筛选无结果时返回 `items: []` 和 `total: 0`，不是错误。
- 设置页 Provider 只读展示；如果没有额外配置，本阶段显示 `mock`。

## 测试策略

后端测试覆盖：

- `q` 能匹配 caption。
- `q` 能匹配 tags。
- `q` 能匹配 file_path。
- `format` 能过滤格式。
- `tag` 和 `q` 可以组合。
- `sort` 和 `order` 返回正确顺序。
- 非法 `sort` 返回 400。
- 非法 `order` 返回 400。
- 现有图库、详情、识别和批量识别测试继续通过。

前端验证覆盖：

- `npm run build` 通过。
- 搜索框能触发请求并刷新列表。
- 筛选无结果时显示空状态和清空入口。
- 排序控件能改变请求参数。
- 设置页显示 Provider 卡片。

## 验收标准

- 用户能通过关键词快速找到图片。
- 用户能按标签和格式缩小结果。
- 用户能按时间、文件大小或尺寸排序。
- 后端对非法排序字段和排序方向返回 400。
- 设置页能清楚显示当前 Provider 为 `mock`。
- 单张识别和批量识别功能不回退。
- 后端测试全部通过。
- 前端构建通过。
