# PicMind Phase 4 设计规格

## 目标

PicMind Phase 4 的目标是把现有 Mock 图片识别能力升级为「可扩展的本地内容识别基础」，让项目沿着后续接入 YOLO 的路线继续发展。

本阶段不直接接入 YOLO 模型，而是先完成 YOLO 前置基础：保持现有 Mock 识别流程稳定，增加本地主色识别，把颜色标签写入现有识别结果，并整理识别结果结构，让后续 YOLO 能自然写入 `objects` 和搜索标签。

## 当前完成度

### 已完成

- 本地图片扫描、SQLite 索引和图片元数据持久化。
- 图库、图片详情、看板和设置页面。
- Mock 图片识别 Provider。
- 单张图片重新识别。
- 批量识别任务创建和状态查询。
- 图库关键词搜索，支持匹配 `file_path`、`caption` 和 `tags`。
- 图库选择操作，支持全选和全选尚未识别图片。

### 已知短板

- 当前识别结果仍主要来自 Mock Provider。
- 搜索栏只能搜已有文本字段，不能搜图片里真实存在但未入库的内容。
- `objects` 字段目前为空，尚未承载真实物体检测结果。
- 未接入 YOLO，因此不能识别人、车、猫、狗等目标。

## 范围

### 包含

- 增加本地主色识别能力。
- 主色识别使用现有 Pillow 依赖读取图片像素。
- 将主色映射为中文颜色标签，例如 `红色`、`黄色`、`蓝色`、`绿色`、`黑色`、`白色`、`灰色`、`橙色`、`紫色`、`粉色`、`棕色`。
- 单张识别和批量识别完成后，把颜色标签保存进 `annotations.tags`。
- 保持现有横图、竖图、方图方向标签。
- 保持初始扫描图片仍标记为尚未识别，继续使用 `待分析` 表示未识别状态。
- 识别完成后移除 `待分析`，避免已识别图片继续被「全选尚未识别」选中。
- 明确 `objects` 的 YOLO 预留结构。
- 增加测试覆盖颜色识别和搜索颜色标签。

### 不包含

- 本阶段不直接运行 YOLO。
- 不要求用户现在提供 YOLO 模型路径。
- 不下载模型、不安装大体积深度学习依赖。
- 不做向量搜索、CLIP、语义搜索或视觉大模型描述生成。
- 不做颜色区域框选或目标位置绘制。
- 不在前端新增复杂识别配置页面。

## 识别结果结构

### tags

`tags` 继续作为面向搜索和筛选的标签列表。

识别完成后的示例：

```json
["本地图片", "landscape", "黄色"]
```

标签规则：

- `本地图片`：保留，表示图片来源。
- `landscape` / `portrait` / `square`：保留，表示图片方向。
- 中文颜色标签：新增，表示图片主色。
- `待分析`：只用于初始扫描后的未识别图片，识别完成后不再保留。

### objects

`objects` 本阶段仍可为空，但结构为 YOLO 预留。

后续 YOLO 接入后的目标结构：

```json
[
  {
    "label": "person",
    "name": "人",
    "confidence": 0.91
  },
  {
    "label": "car",
    "name": "汽车",
    "confidence": 0.84
  }
]
```

字段含义：

- `label`：模型原始英文标签。
- `name`：用于展示和搜索的中文名称。
- `confidence`：识别置信度。

YOLO 接入后，中文 `name` 也应写入 `tags`，让现有搜索栏可以直接搜索物体名。

## 后端设计

### 主色识别

新增一个小型颜色分析单元，职责是：

1. 打开图片文件。
2. 缩小图片以减少计算量。
3. 转换为 RGB。
4. 统计代表性像素颜色。
5. 根据 RGB 距离映射到最近的预设中文颜色。
6. 返回一个中文颜色标签。

颜色识别应该是确定性的：同一张图片每次识别得到相同结果。

### MockRecognizer 扩展

现有 `MockRecognizer` 继续负责返回 `RecognitionResult`。

本阶段扩展后，它返回：

- `caption`: 继续为 `待分析的本地图片`。
- `tags`: `本地图片` + 方向标签 + 主色标签。
- `objects`: 继续为空列表。
- `model_used`: 继续为 `mock`。

虽然 `model_used` 仍为 `mock`，但它已经包含一个真实本地分析能力：主色识别。

### RecognitionService

`RecognitionService` 继续负责：

- 查找图片记录。
- 检查图片文件存在。
- 调用识别器。
- 写入或覆盖 `Annotation`。

本阶段不改变它的外部调用方式，避免影响单张识别和批量识别 API。

### 搜索能力

后端现有 `/api/images?q=...` 已匹配 `Annotation.tags`。

因此颜色识别结果写入 tags 后，搜索颜色无需新增搜索 API。用户输入 `黄色`、`蓝色`、`黑色` 等关键词时，现有查询即可命中。

## 前端设计

本阶段前端不新增复杂页面。

图库页继续使用现有搜索框。用户完成识别后，可以直接搜索颜色词：

- `黄色`
- `蓝色`
- `红色`
- `黑色`
- `白色`

批量识别流程不变。识别完成后刷新列表时，图片卡片展示的标签里会出现颜色标签。

## YOLO 后续接入路径

本阶段结束后，下一阶段接入 YOLO 时不需要推翻当前结构。

后续步骤会是：

1. 增加 YOLO 识别器实现。
2. 读取用户提供的 YOLO 模型文件路径，例如 `yolov8n.pt` 或 `yolov8s.pt`。
3. 对图片运行目标检测。
4. 将检测结果写入 `objects`。
5. 将中文物体名写入 `tags`。
6. 搜索栏继续通过 `tags` 搜到物体。

这样 YOLO 是现有识别管线的扩展，而不是重做项目。

## 错误处理

- 图片文件不存在时，继续沿用现有 `ImageFileMissingError`。
- 图片文件无法被 Pillow 打开时，识别应失败并暴露错误，不静默写入错误标签。
- 无法判断主色时，不写入颜色标签，但识别流程仍应返回方向标签和基础标签。
- 搜索无结果时继续返回空列表，不视为错误。

## 测试策略

后端测试覆盖：

- 主色识别能把红色测试图片识别为 `红色`。
- 主色识别能把黄色测试图片识别为 `黄色`。
- MockRecognizer 返回基础标签、方向标签和颜色标签。
- RecognitionService 保存颜色标签到 `Annotation.tags`。
- `/api/images?q=黄色` 能通过 tags 搜到黄色图片。
- 识别完成后的 tags 不包含 `待分析`。
- 现有单张识别、批量识别、图片列表和设置接口测试继续通过。

前端验证覆盖：

- `npm --prefix frontend run build` 通过。
- 图库搜索框输入颜色词后能展示匹配图片。
- 全选尚未识别不会选中已经识别并去除 `待分析` 的图片。

## 验收标准

- 用户能识别图片并看到颜色标签。
- 用户能用搜索栏搜索中文颜色词找到图片。
- 新加入但未识别的图片仍被视为尚未识别。
- 已识别图片不会继续出现在「全选尚未识别」结果里。
- `objects` 字段结构已经为 YOLO 接入明确预留。
- 本阶段不要求 YOLO 模型路径，不运行 YOLO。
- 后端测试通过。
- 前端构建通过。
