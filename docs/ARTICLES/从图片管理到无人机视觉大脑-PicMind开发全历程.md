# 从图片管理到无人机视觉大脑：PicMind 开发全历程

> 本文记录了 PicMind 项目从一个简单的本地图片管理工具，逐步演进为具备完整实时视觉能力的无人机感知系统的全过程。

---

## 🎯 项目愿景：在 PC 上开发，在无人机上运行

PicMind 的核心理念很简单：

> **在 PC 上调试好的每一行视觉代码，都可以零改动地迁移到无人机机载电脑上运行。**

我们不追求"做一个玩具 demo"，而是按照真实机载系统的标准来设计每个模块。从接口抽象到错误处理，从性能节流到资源互斥，每一步都在为真正的飞行做准备。

---

## 🚀 开发历程全记录

### 第一阶段：静态图片识别（基础能力）

**目标：让电脑"看懂"图片里有什么**

| 功能模块 | 实现要点 | 无人机关联 |
|---------|---------|-----------|
| **图片扫描与索引** | 递归扫描本地目录，提取 EXIF 元数据 | 无人机机载录像回放分析 |
| **YOLOv8 物体检测** | 集成 Ultralytics YOLO，输出 label/confidence/bbox | 无人机视觉感知的基础 |
| **边界框渲染** | 提取 BboxOverlay 组件，可复用 | 目标可视化、地面站显示 |
| **场景分类** | 从投票制优化到优先级制，室内/室外判断 | 自动切换飞行参数模式 |
| **危险目标检测** | 16 类 COCO 危险标签（人、车、动物） | 避障系统输入 |

**这个阶段结束后**：系统可以批量处理本地图片，识别内容，并标记潜在危险。

---

### 第二阶段：实时摄像头流（从静态到动态）

**目标：让视觉系统"活起来"，处理连续视频帧**

| 功能模块 | 实现要点 | 无人机关联 |
|---------|---------|-----------|
| **WebSocket 推流** | 后端编码 JPEG + base64，前端实时渲染 | 无人机 → 地面站视频传输协议 |
| **单客户端互斥锁** | `asyncio.Lock` 保护摄像头资源 | 避免多设备争抢机载摄像头 |
| **推理节流** | 每 5 帧跑一次 YOLO，中间帧复用结果 | 适配机载 CPU/GPU 性能限制 |
| **摄像头生命周期管理** | open/read/close 完整封装，异常安全 | 摄像头热插拔、故障恢复 |
| **LivePreview 页面** | React + WebSocket，状态机管理 | 地面站实时监控界面原型 |

**这个阶段结束后**：系统从"批处理工具"变成了"实时视觉引擎"。

---

### 第三阶段：中心点偏移（从"看到"到"控制"）

**目标：让视觉系统的输出可以直接用来控制飞行！**

这是一个里程碑式的跨越——系统不再只是输出"看到了什么"，而是开始输出"目标在哪里，我该怎么动"。

#### 核心设计

**新增数据契约字段：**

```json
{
  "target_offset": {
    "label": "person",
    "name": "人",
    "confidence": 0.91,
    "target_center": {"x": 0.4, "y": 0.5},
    "dx": -0.1,
    "dy": 0.0,
    "dx_px": -64,
    "dy_px": 0
  }
}
```

**前端可视化元素：**

| 元素 | 设计 | 含义 |
|------|------|------|
| 🟦 中心十字 | 青色，永远在画面中心 | 瞄准参考点 |
| 🟡 目标圆点 | 黄色，在 bbox 中心 | 当前锁定目标 |
| ➡️ 偏移箭头 | 黄色，从中心指向目标 | 方向指示 |
| 📊 数值显示 | 左上角，颜色编码 | 绿=已居中，黄=轻微偏移，红=明显偏移 |

#### 目标选择策略

**只追踪危险目标**——这是关键的设计决策！

椅子、桌子、书本这些背景物体不应该干扰飞控。我们只关心：
- 人、汽车、摩托车、自行车、卡车、公交车
- 鸟类、猫、狗、马、羊、牛、大象、熊、斑马、长颈鹿

**这完全符合无人机避障逻辑：只需要躲开会动的东西。**

#### 飞控映射表

现在系统输出已经可以直接控制无人机了：

| 输出值 | 飞控动作 |
|--------|---------|
| `dx_px < -50` | 🎮 右转 |
| `dx_px > +50` | 🎮 左转 |
| `dy_px < -50` | 🎮 抬头 |
| `dy_px > +50` | 🎮 低头 |
| `\|dx\| < 20 AND \|dy\| < 20` | ✅ 目标居中，锁定！ |

**这个阶段结束后**：系统具备了基础的目标追踪和转向指示能力。

## 📐 架构设计亮点

### 1. 面向接口的模块抽象

每个核心服务都遵循依赖倒置原则：

```python
# LivePipeline 不直接依赖具体摄像头实现
def __init__(self, camera: CameraProtocol, recognizer: RecognizerProtocol):
```

**迁移价值**：PC 上用 `cv2.VideoCapture`，无人机上换成 MAVLink 摄像头驱动，**一行代码都不用改**。

### 2. 状态机驱动的前端

LivePreview 页面用清晰的状态管理：

```
idle → connecting → running → error
  ↓          ↓          ↓        ↓
停止       连接中    实时流    错误处理
```

**迁移价值**：真实地面站软件也是同样的状态机设计。

### 3. 单一职责组件复用

`BboxOverlay` 同时服务于：
- 静态图片详情页（离线分析）
- 实时预览页面（在线监控）

**迁移价值**：组件设计可以直接搬到地面站 UI 里。

## 🗺️ 后续路线图

```
已完成 → H3 多目标追踪 → H2 距离估算 → 真实飞控集成
  ✅          ⏳               ⏳               📋
```

### H3：多目标追踪 & ID 保持
- 给每个检测到的目标分配稳定的唯一 ID
- 使用 BoT-SORT 或 ByteTrack 算法
- 即使目标被短暂遮挡再出现，ID 保持不变
- **无人机能力**："锁定 1 号目标，别跟丢了！"

### H2：距离估算
- 相机标定 + bbox 高度 → 推算真实世界距离
- 卡尔曼滤波平滑距离输出
- **无人机能力**："人在前方 5 米，以 1.2 m/s 移动"

### 真实飞控集成
- 通过 MAVLink 协议连接 PX4/ArduPilot
- 把 `dx_px/dy_px` 转成 RC 通道输出
- 实现完整的"视觉追随"模式

## 💡 开发感悟

### 1. "先在地面跑通"原则

每一个视觉算法，都先在 PC 上验证成熟：
- 单元测试全部通过
- 手工验证各种边界场景
- 性能参数调优到满意

然后再搬到无人机上——机载调试的成本是地面的 10 倍！

### 2. 测试是飞行安全的底线

177 个单元测试不是为了好看，而是为了：
- 改代码时不破坏已有功能
- 边缘场景（空输入、模糊画面、半遮挡）都能正确处理
- 机载环境出问题时，可以快速在本地复现定位

### 3. 架构比算法更重要

YOLO 是现成的，但如何把 YOLO 包装成：
- 可复用的服务
- 可测试的模块
- 可扩展的接口
- 容错的数据流

这才是从"demo"到"产品"的关键跨越。
